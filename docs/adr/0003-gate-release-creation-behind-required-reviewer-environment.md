# Gate release creation behind a required-reviewer GitHub Environment, in every repo

Every repo's release-creation step, the job that runs `gh release create` and, for `tsconfig`, publishes the package, runs behind a required-reviewer GitHub Environment. The four repos consumed as composite actions (`action-setup-workspace`, `action-validate-release-tag`, `action-extract-release-notes`, `action-create-github-release`) use an environment named `release`. `tsconfig` keeps its existing `package-publish` environment (see its own ADR 0004/0005), which already gated its npm-publish step for unrelated reasons and now serves this purpose too. A human approves the job after `validate` passes and before anything release-visible happens.

This supersedes the "ungated" decision in `action-setup-workspace`'s own [ADR 0002](https://github.com/dnd-mapp/action-setup-workspace/blob/main/docs/adr/0002-automated-release-and-discussion-creation.md), which argued a gate wasn't worth a second manual click since the tag push was already the approval step. That reasoning is deliberately overridden platform-wide: consistency across repos, and a checkpoint immediately before an irreversible, externally-visible action (a GitHub Release, a Discussion, and for `tsconfig` a published package), outweighs the convenience of skipping it.

# The three release-tooling repos fully cross-consume each other for their own releases

`action-validate-release-tag`, `action-extract-release-notes`, and `action-create-github-release` (see `action-setup-workspace`'s [ADR 0003](https://github.com/dnd-mapp/action-setup-workspace/blob/main/docs/adr/0003-extract-release-tooling-into-independent-actions.md) for why they exist) each own exactly one release-tooling concern, but none owns all three concerns its own release needs. Each repo's own `release.yml` uses `uses: ./` only for the concern it owns, and calls the other two externally, pinned to a commit SHA, for the concerns it doesn't. This is the same shape `action-setup-workspace` and `tsconfig` use to consume all three.

The alternative, each repo keeping small inline/duplicated logic for the concerns it doesn't own rather than depending on its siblings, was rejected: it would reintroduce, inside the very repos that exist to eliminate it, the duplicated validate/extract/create logic this extraction was meant to kill. The bootstrapping-circularity concern that originally justified a strict build order (`action-validate-release-tag` -> `action-extract-release-notes` -> `action-create-github-release`) no longer applies once all three are tagged; any of the three can consume any other today.
